# План тестирования

Этот документ описывает стратегию тестирования для RAG-приложения.

## 1. Модульное тестирование

### Цель

Проверить корректность работы отдельных компонентов (функций и классов) в изоляции.

### Инструменты

*   `pytest` - для написания и запуска тестов.
*   `pytest-mock` - для создания мок-объектов.

### Тестовые случаи

*   **`QueryEmbeddingService`**:
    *   Проверить, что кэширование работает (второй вызов с тем же запросом не вызывает `model.encode`).
    *   Проверить, что векторизация вызывается для новых запросов.
*   **`ContextRetriever`**:
    *   Проверить, что кэширование работает (второй вызов с теми же chunk ID не выполняет запрос к БД).
    *   Проверить, что корректно извлекается контекст из мок-БД.
*   **`RAGPipelineOrchestrator`**:
    *   Проверить, что оркестратор правильно вызывает все компоненты.
    *   Проверить, что возвращается сообщение о пустом контексте, если он не найден.
*   **Логика уточняющих вопросов в `api.py`**:
    *   Проверить, что при низкой оценке возвращается "уточните вопрос".
    *   Проверить, что счетчик уточнений сбрасывается при успешном ответе.
    *   Проверить, что после трех неудачных попыток возвращается ответ с низкой уверенностью.

## 2. Интеграционное тестирование

### Цель

Проверить взаимодействие между компонентами системы, включая базу данных, Redis и API.

### Инструменты

*   `pytest`
*   `docker-compose` - для запуска тестового окружения с PostgreSQL и Redis.
*   `httpx` - для выполнения запросов к API.

### Тестовые случаи

*   **Полный цикл RAG**:
    *   Отправить запрос к эндпоинту `/query`.
    *   Проверить, что в ответе содержится сгенерированный текст и ссылка на источник.
    *   Проверить, что история диалога сохраняется в Redis.
*   **Проверка контекста**:
    *   Отправить два связанных запроса и убедиться, что второй ответ учитывает контекст первого.
*   **Обработка ошибок**:
    *   Отправить некорректный запрос и проверить, что API возвращает ошибку 500.

## 3. Нагрузочное тестирование

### Цель

Оценить производительность и стабильность приложения при большом количестве одновременных пользователей.

### Инструменты

*   `locust` - для создания и симуляции нагрузки.

### Сценарии

*   **Базовый сценарий**:
    *   Симулировать 100 одновременных пользователей, отправляющих запросы к `/query`.
    *   Измерить среднее время ответа и количество ошибок.
*   **Сценарий с кэшированием**:
    *   Повторно отправить те же запросы, чтобы оценить влияние кэширования.
    *   Сравнить результаты с базовым сценарием.

Этот план тестирования поможет обеспечить высокое качество и надежность вашего приложения.
